II - Modélisation

Pour créer une page pour s'identifier vers la connexion à un compte il faut une base de données ou Database

Avant de créer une BDD(base de données), il est essentiel de se poser et de réflechir à la modélisation de la base : pour éviter de se tromper.

Changer le modélisation d'une BDD en cours de projet n'est pas récommandé; cela nous forcera à réécrire une bonne partie du code

- Comment modéliser une BDD?

On peut modéliser une BDD sur une feuille de papier ou on peut utiliser un logiciel pour nous aider. Exemple : MySQL Workbench

Pour une bonne conception de BDD, il faut réflechir en terme de sujets en modélisant ceux du monde réel. La modélisation se compose de différents tables (une table de sujet); ex. si nous vendons des produits il faut une table des produits et une table des commandes

III- Les tables

  • 1 sujet représente 1 table dans une base de données.
  • 1 table est un emplacement de sauvegarde.
  • Les données sont stockées à l'intérieur de tables. Une table peut être comparée à une liste, qui contient des enregistrements relatifs à un sujet bien défini.

Il faut réfléchir au sujet et à ses conséquences. Si nous vendons des produits, nous aurons certainement des membres. 1 table membre sera donc nécessaire. En terme de fonctionnalité, si nous souhaitons proposer un abonnement newsletter aux membres, nous aurons besoin d'1 table newsletter repertoriant les membres abonnés
table = 1 sujet

IV- Les colonnes ou les champs

Chaque table possède généralement plusieurs champs (aussi appelés colonnes). Les colonnes / champs représentent des caractéristiques relatives au sujet (la table).

Pour savoir, quelle colonne mettre dans quelle table, il faut se poser la question suivante : qu'est-ce qui pourrait décrire mon sujet ? quelles sont les informations sur mon sujet ?
Par exemple, la table membre aura les champs : pseudo, mot de passe, nom, prenom, adresse, email, etc.

V- Les types de champ

Type de champ Description Contexte
VARCHAR Chaîne (jusqu'à 255 caractéres) Pour enregistrer un pseudo, un email, le titre d'un produit ou d'un article de blog
TEXT Chaîne de caractéres (illimité) Pour un contenu textuel long, un article de blog
INT Numérique Pour un identifiant, un prix, un code postale le nombre de produits en stocks etc.
DATE Pour la date d'un enregistrement (une inscription), la date d'une commande etc.
ENUM Une liste (une énumération) Pour proposer une liste de termes ex.:"Madame","Monsieur"

VI- Types de données PK AI pour les ID

a/ PK - Primary Key, clef primaire

  • Les identifiants sont des champs un peu particuliers car ils ne décrivent pas le sujet, ce sont des champs systématiquement présents dans chaque table et ce en 1ére position
  • Les identifiants sont uniques
  • chaque table posséde un champs identifiant, nous appelons cela une clef primaire (PK)
  • Par ex. nous pourrons appeler le champs "id" ou encore "id_membre" ou "idMembre" "id_user" pour la table "membre" ou "user"

b/AI - Auto Increment, auto incrémenté

  • Auto incrémenté est une option permettant de générer un numéro unique dans un champ de type clef primaire (identifiant).
  • Par conséquent, retenez bien que le 1er champ de chaque table sera systématiquement un "id" qui sera PK et AI.

c/ NULL, NOT NULL

  • Dans chaque champ, nous pourrons indiquer si nous acceptons les valeurs NULL ou NOT NULL
  • NULL est un type de valeur en informatique évitant de laisser un champ vide si nous n'avons pas d'informations à y déposer

d/ Les relations

Il arrive parfois que le sujet interagissent entre eux; exemples

  • 1 membre commande un produit (ou 1 produit est commandé par un membre), nous devons enregistrer l'information dans une table commande
  • conducteur conduit véhicule (ou 1 véhicule est conduit par conducteur), il nous faudra une table permettant de préciser qui conduit quel véhicule

VII- Les cardinalités

  • Les cardinalités permettant de connaître le chiffre minimum et maximum d'enregistrements pour une relation
  • Example 1: 1 homme est le fils d'une femme et d'une seule (minimum = 1 et max =1)
    En revanche une femme peut n'avoir aucun enfant ou plusieurs (min = 0 et max = n)
  • Exemple 2 : 1 livre possède forcément 1 auteur et qu'1 seul. Relation (1,1). Minimum 1 auteur, Maximum 1 auteur. 1 auteur peut avoir écrit plusieurs livre (ou aucun). Relation (0,n). Minimum 0 livre, Maximum N livres.

VIII- Les tables de jointure

Une table de jointure permet de faire le lien entre 2 tables

Exemple une société de taxi, qui posséde des conducteurs et des véhicules.

  • 1 conducteur peut conduire 0 ou plusieurs véhicules
  • 1 véhicule peut être conduit par 0 ou plusieurs conducteurs

Comment savoir quel conducteur conduit quel véhicule ? ou dans l'autre sens quel véhicule est conduit par quel conducteur?

Avec cette table de jointure, nous savons qu'Alexandre est autorisé à conduire la BMW et la Mercedes et Julien ne peut conduire que la BMW

IX- Clefs étrangères ou FK

Les clefs étrangères sont des champs forcément clefs primaires dans leur table d'origine et qui se retrouvent dans une table extérieur.

Un champ clef étrangère permet de faire la relation avec un autre sujet.

X- Les enregistrements

Chaque enregistrement représente 1 ligne dans la table de la BDD. Chaque nouvel enregistrement va s'auto-incrémenter. Même si un ID est supprimé, il ne sera pas réutilisé !

XI- Comment interroger la BDD

a/ Les requêttes

Une requêtte permet de poser une question afin d'obtenir une réponse, ou encore de donner un ordre.

Nokus terminerons toutes nos requête SQL par un point-virgule ";" pour que MySQL sache que nous avons terminé d'écrire et qu'il doit exécuter notre demande.

De manière standard les requêtes SQL seront écrites en majuscules SELECT * FROM membre;

b/ Types de requêtes

Il existe 4 types de requêtes :

  • Sélection SELECT; requête question/réponse, nous demandons et obtenons une réponse
  • Insertion INSERT; requête d'action, qui aura un impact sur les données
  • ModificationUDATE;requête d'action, qui aura un impact sur les données
  • suppressionDELETE;requête d'action, qui aura un impact sur les données
